# EasyMeeting 面试深挖问答（16题，工程化详解版）

> 版本基线：截至 2026-05-05 当前实现。
> 目标：让没看过项目的人，也能通过回答理解架构、流程、边界和演进思路。
> 结构：每题都包含「知识点」「项目落地」「代码细节」「工程思维」「追问应对」。

## 1. 你这个项目的整体架构是怎样的？

### 知识点
实时会议系统可以拆成两条主链路：
- 媒体链路：音视频数据传输，决定清晰度、延迟和稳定性。
- 控制链路：登录、入退会、信令、主持人操作、聊天、邀请、文件、后台管理。

EasyMeeting 当前重点在控制链路，媒体链路先使用客户端直连方式，服务端主要做“状态中心 + 消息调度中心”。

### 项目落地
项目是单体分层架构，但具备横向扩展能力：
- 接口层：负责会议创建、预入会校验、邀请、联系人、管理后台、版本更新、文件接口。
- 实时层：基于 Netty 管理 WebSocket 长连接，承接 WebRTC 信令透传。
- 会话层：Redis 保存验证码、token 会话、用户当前会议、会议在线成员和心跳状态。
- 数据层：MySQL 保存会议、成员、邀请、文件、聊天消息等持久数据。
- 消息层：保留 MessageHandler 抽象，但默认且唯一跨节点实现是 RabbitMQ。

### 代码细节（按层）
- 入口先区分 HTTP 和 WebSocket 两条路径。
- HTTP 请求进入 Controller，再进入 Service 做权限和状态校验，最后落 DB 或写缓存。
- WebSocket 消息进入 Netty pipeline，先鉴权再解析，封装成统一消息对象，交给消息总线。
- 消息在本机或其他节点被消费后，按“单播用户”或“群播房间”下发。

### 工程思维
- 容量：媒体不进业务后端，先保证控制链路可扩容。
- 复杂度：单体分层比微服务更容易快速交付和排障。
- 可演进：消息总线固定使用 RabbitMQ，为后续集群化留好口。
- 成本：先交付核心功能，再针对热点瓶颈做定向升级。

### 追问应对
- 被问“是不是微服务”：不是，当前是单体分层 + 实时节点可扩容。
- 被问“为什么不先拆服务”：会议系统先要把实时链路稳定性打透，过早拆分会放大治理成本。

## 2. 为什么一开始选 WebRTC Mesh，而不是 SFU？

### 知识点
Mesh 的优势是快：客户端之间直接协商，服务端不承担媒体转发。
Mesh 的代价是规模：连接和上行压力随人数显著增长，会议越大越吃力。

### 项目落地
项目初期优先目标是中小会可用性和迭代速度：
- 先把“会控能力”做稳：鉴权、入会、主持人控制、邀请、成员状态同步。
- 媒体侧先走 Mesh，降低首期基础设施复杂度。
- 通过信令中心化设计，为后续平滑引入 SFU 预留空间。

### 代码细节（按流程）
- 客户端发起 Peer 信令，服务端只负责鉴权、解析、路由转发。
- 服务端不做音视频编解码，不做媒体转发，只维护会议信息和消息分发。
- 这样可以把媒体复杂度和业务复杂度隔离开。

### 工程思维
- 价值排序：先做“可用”和“可维护”，再做“超大规模”。
- 技术债可控：信令中心化后，升级 SFU 不需要推翻业务层。
- 成本控制：减少早期媒体基础设施投入，把资源集中在核心功能闭环。

### 追问应对
- 被问“Mesh 能撑多少人”：没有固定值，取决于终端性能、码率、网络质量。
- 被问“什么时候切 SFU”：当实际会议规模和弱网指标持续超阈值，就该切换。

## 3. Mesh 的瓶颈在哪？怎么升级到 SFU？

### 知识点
Mesh 的三个核心瓶颈：
- 上行带宽：每个用户要发给多个对端。
- 端侧负载：编码解码和协商压力叠加。
- 控制复杂度：成员频繁变化会触发大量协商。

### 项目落地
当前项目已经具备“控制与媒体解耦”的升级前提：
- 会议状态机、权限模型、邀请流程都在业务层。
- 信令路由独立，不和具体媒体转发方式耦合。
- 升级时主要改“协商目标”和“媒体路径”，业务逻辑可复用。

### 代码细节（升级路径）
- 第一步：保留现有会控 API 和状态模型。
- 第二步：新增 SFU 路由策略，把 Peer 协商目标从“用户到用户”切为“用户到 SFU”。
- 第三步：保留双轨开关，按会议规模或会议类型决定走 Mesh 还是 SFU。

### 工程思维
- 渐进演进：先做试点房间，不一次性全量切换。
- 风险隔离：保留回滚开关，失败时可回到旧路径。
- 可观测先行：升级必须绑定关键指标门禁，不靠主观体验决策。

### 追问应对
- 被问“会不会重写很多代码”：不会，主要是媒体路径变化，控制层可复用。
- 被问“最大风险是什么”：信令和媒体状态不同步，因此要先补足观测和灰度机制。

## 4. NAT 穿透（ICE/STUN/TURN）在项目里怎么落地？

### 知识点
ICE、STUN、TURN 的主职责通常在客户端 WebRTC 栈：
- STUN 帮助发现可达地址。
- TURN 用于直连失败时中继。
- 服务端业务层主要负责信令交换，不负责生成候选。

### 项目落地
项目后端明确定位为“信令转发层”：
- 服务端接收客户端上传的信令类型和信令数据。
- 服务端做鉴权、路由、分发，不做 STUN/TURN 策略管理。
- 客户端通过 RTCConfiguration 管理候选策略和 TURN 使用。

### 代码细节（数据怎么走）
- 客户端发送信令 JSON（含 signalType/signalData/目标信息）。
- 服务端解析为信令 DTO，再封装成统一消息对象。
- 消息对象通过 RabbitMQ 广播到节点后，再由本地 WebSocket 单播或群播到目标端，目标端继续完成协商。

### 工程思维
- 职责边界清晰：媒体连通策略留给 RTC 层，业务层不做重复治理。
- 低耦合：避免把 TURN 策略硬编码进业务服务。
- 可观测补偿：业务层可接入客户端上报，做 TURN 使用率和失败率统计。

### 追问应对
- 被问“后端为什么不接管 TURN”：因为那会增加耦合和维护成本，且收益有限。
- 被问“怎么排查穿透问题”：通过客户端统计 + 服务端信令链路日志联合定位。

## 5. 直连失败或 TURN 压力高时，怎么降级？

### 知识点
实时系统降级原则通常是“先保可用，再保画质”：
- 优先保语音。
- 再降视频分辨率/帧率。
- 必要时只保控制链路和文本链路。

### 项目落地
当前后端已经具备控制面降级能力：
- 支持成员媒体状态更新并广播。
- 即便媒体弱化，会议管理、聊天、邀请、主持人控制仍可持续。
- TURN 资源调度属于 RTC 基础设施侧，后端当前不直接管理。

### 代码细节（控制面保证）
- 会议成员媒体状态放在运行态缓存里，变更后立即广播房间。
- 客户端据此更新 UI 和会中状态，不依赖重拉全量。
- 关键会控消息和成员状态由服务端统一下发，保证会议信息不失联。

### 工程思维
- 分层兜底：媒体降级不应拖垮会控。
- 用户体验：即便视频降级，会议秩序和沟通链路必须在。
- 演进方向：后续可增加自动策略下发和弱网告警闭环。

### 追问应对
- 被问“有没有自动降级”：当前以客户端策略为主，后端有可扩展位。
- 被问“最坏情况怎么保”：至少保证语音或文本和会控链路可用。

## 6. Netty 线程模型和 Pipeline 怎么设计？

### 知识点
Netty 常见模型是 boss/worker：
- boss 负责连接接入。
- worker 负责读写和业务处理。

Pipeline 是可组合责任链，按协议顺序处理。

### 项目落地
当前处理顺序清晰：
- HTTP 编解码。
- HTTP 聚合。
- 空闲检测。
- 心跳处理。
- token 鉴权。
- WebSocket 协议处理。
- 业务消息处理。

### 代码细节（每一步做什么）
- 先完成 HTTP Upgrade 前置处理，确保握手合法。
- 空闲检测发现读超时后触发连接关闭，避免僵尸连接长期占用资源。
- 鉴权放在业务处理前，非法连接尽早拒绝。
- 业务 handler 只处理已鉴权的文本帧，逻辑更纯粹。

### 工程思维
- 性能：把重逻辑放在后面，前面先做低成本过滤。
- 安全：提前拦截非法 token，减少攻击面。
- 稳定性：心跳 + 空闲回收，避免连接泄漏。

### 追问应对
- 被问“为什么先 HTTP 再 WS”：WebSocket 握手本质就是 HTTP Upgrade。
- 被问“为什么要 pipeline 分层”：每层职责单一，便于定位问题和独立调优。

## 7. 为什么用 Netty 做信令，而不是直接用 Spring WebSocket？

### 知识点
Spring WebSocket 更偏框架友好，Netty 更偏底层可控。
在实时长连接场景，细粒度控制连接生命周期非常关键。

### 项目落地
项目选择 Netty 的核心原因：
- 更细粒度控制握手、心跳、空闲超时、关闭行为。
- 更容易和自定义消息总线、房间路由机制低开销整合。
- 对大规模连接的行为更可预测。

### 代码细节（控制点）
- 自定义 handler 可以直接操作 channel 上下文和属性。
- 房间广播与单播可以直接复用 Netty channel/channelGroup。
- 心跳和断线时序完全由服务端掌控。

### 工程思维
- 取舍：牺牲一部分开发便利，换取实时链路可控性。
- 运维：可控的连接生命周期，能降低线上不确定行为。
- 成本：后期调优空间更大，避免“框架黑盒”。

### 追问应对
- 被问“Spring WebSocket 能不能做”：能做，但在细粒度控制上不如 Netty。
- 被问“复杂度会不会提高”：会，但复杂度换来的是链路稳定和可调优。

## 8. 一条 WebRTC 信令消息在系统里怎么走？

### 知识点
信令链路本质是“鉴权 -> 解析 -> 路由 -> 下发”。
重点不是计算媒体内容，而是保证正确送达和状态同步。

### 项目落地
当前链路时序非常明确：
1. 客户端带 token 建立 WebSocket。
2. 服务端从连接参数校验 token，并把 userId 绑定到连接上下文。
3. 收到文本帧后解析为信令 DTO。
4. 封装统一消息对象，补充 messageId 和 sendTime。
5. 交给消息总线实现跨节点同步。
6. 消费端根据是否有目标用户决定单播或群播。

### 代码细节（对象传递）
- 输入对象：WebSocket 文本帧。
- 解析对象：Peer 信令 DTO。
- 中间对象：统一 MessageSend DTO（含消息类型、目标范围、内容、时间戳）。
- 输出对象：目标端收到的文本消息。

### 工程思维
- 抽象统一消息对象，屏蔽不同消息总线差异。
- 单播和群播走同一发送抽象，减少分叉逻辑。
- 通过 messageId/sendTime 给客户端排序和幂等提供依据。

### 追问应对
- 被问“怎么区分单播群播”：看是否有目标用户标识。
- 被问“跨节点如何送达”：依赖消息总线广播，消费后本地路由到连接。

## 9. 怎么保证消息一致性和可靠性？

### 知识点
实时系统一般做“分层一致性”：
- 传输层：ACK、重试。
- 业务层：状态机、幂等、补偿。
- 存储层：关键状态持久化。

### 项目落地
当前项目“已落地”的能力：
- RabbitMQ 发送端启用 publisher confirm，消费端使用手动 ACK。
- 失败按 retryCount 处理，最多重试 3 次，超过后进入死信队列。
- 发送前补 messageId/sendTime，并在消费侧用 messageId 做幂等去重。
- 邀请链路采用先落库后发消息，并对待处理邀请去重。

当前“尚未落地”的能力：
- 没有全链路 outbox。
- 已补充 RabbitMQ DLX/DLQ，并提供管理员死信重试入口。
- 消费侧已按 messageId 做幂等去重，但还没有覆盖所有业务写操作的 outbox/幂等表。

### 代码细节（失败处理）
- 消费异常时读取消息头的重试计数。
- 重试次数未达阈值则重新投递。
- 超过阈值后进入死信队列，并记录 meeting_event_log。
- 语义上属于至少一次，而非 exactly-once。

### 工程思维
- 现实取舍：先保证核心链路可用，再补全“强一致配套”。
- 边界清晰：明确告诉面试官当前是至少一次语义。
- 演进路径：outbox + 会议内 sequence + 更完整的业务幂等键。

### 追问应对
- 被问“能不能保证绝对不重复”：不能保证 exactly-once；当前依赖 messageId 去重和业务状态收敛。
- 被问“为什么不一次做到位”：一致性体系建设成本高，需按业务价值分阶段投入。

## 10. 怎么处理消息有序性？

### 知识点
分布式实时系统里，全局强顺序代价很高。
工程上常见方案是“局部有序 + 状态收敛”。

### 项目落地
当前策略偏工程实用：
- 通过统一广播链路降低乱序概率。
- 消息携带时间和标识，供客户端排序参考。
- 不做全局单调序列号，不承诺全局强顺序。

### 代码细节（冲突如何处理）
- 关键控制事件发生时，服务端先更新成员状态，再广播事件。
- 客户端收到事件后，以最新状态为准刷新视图。
- 即便消息先后出现抖动，最终渲染通过状态机收敛。

### 工程思维
- 先保证“最终正确”，再追求“严格顺序”。
- 强顺序需要额外代价，只有在关键场景再引入会议内 sequence。
- 有序性设计必须和幂等设计一起做。

### 追问应对
- 被问“踢人和聊天同时发生怎么办”：以成员状态和在会关系为准。
- 被问“如何增强”：按会议维度加递增序列和版本号。

## 11. 聊天消息为什么按 meetingId 做 32 分表？

### 知识点
聊天消息是典型高写入场景，分表主要解决：
- 单表膨胀导致索引变慢。
- 写入竞争导致延迟抖动。
- 查询性能随数据量上升而退化。

### 项目落地
项目采用 meetingId 哈希路由 32 张逻辑表：
- 使用 murmurHash 做路由值。
- 取模定位目标分表。
- 服务层统一计算表名，业务逻辑对分表无感知。

### 代码细节（如何透明）
- 业务层只传 meetingId 和消息内容。
- 路由工具先算出目标表。
- Mapper 层按目标表执行插入或查询。

### 工程思维
- 性能：会议内查询天然命中单表。
- 稳定：避免单表过热。
- 风险：分表数量固定，后续扩容和迁移成本需要提前设计。

### 追问应对
- 被问“为什么 32”：在当前规模下平衡分散度与运维复杂度。
- 被问“会不会倾斜”：会，热点会议仍可能打到单分表，需要专项治理。

## 12. 分表后跨会议查询、扩容和热点治理怎么做？

### 知识点
分表后的三个典型问题：
- 跨分片聚合复杂。
- 固定分片数扩容困难。
- 热点 key 仍可能形成热点分片。

### 项目落地
当前系统优先保障在线业务主路径：
- 主路径是“按会议查询”，不是全局报表。
- 跨会议统计不走在线实时聚合。
- 先保证会中读写性能和稳定性。

### 代码细节（当前边界）
- 现有实现没有内置跨 32 表的实时聚合引擎。
- 扩容到更多分片需要迁移方案支持，不是直接改常量即可。
- 热点治理需要结合缓存和限流策略，而不是只依赖哈希路由。

### 工程思维
- 在线与离线分治：在线库负责交易和实时，会后分析走离线链路。
- 演进建议：双写迁移、回放校验、灰度切流。
- 目标不是“所有查询都快”，而是“关键路径稳定快”。

### 追问应对
- 被问“现在能跨表聚合吗”：能做，但不经济，不建议作为在线主路径。
- 被问“不停机扩容怎么做”：双写 + 校验 + 灰度，最终切流。

## 13. 权限模型怎么设计？主持人控制怎么落地？

### 知识点
会议权限建议按三层设计：
- 系统层：管理员权限。
- 会议层：主持人和普通成员。
- 状态层：是否正常在会、是否踢出、是否拉黑。

### 项目落地
当前实现是三层叠加：
- 管理后台接口通过全局拦截器校验管理员。
- 会议主持人由会议创建者判定。
- 成员状态驱动权限：被踢出或拉黑后不可继续正常参与。

### 代码细节（权限链路）
- 请求进入后先做登录和管理员校验。
- 服务层再做会议级权限判断，例如只有创建者能踢人、拉黑、结束会议。
- 入会前同时检查数据库状态和运行态缓存状态，防止状态不一致。

### 工程思维
- 权限校验下沉到服务层，避免只靠入口拦截。
- 状态即权限，把会中控制和状态机绑定，减少“逻辑绕过”。
- 已有能力偏核心会控，细粒度角色可后续扩展。

### 追问应对
- 被问“被拉黑还能进吗”：不能，入会前会双层检查并拒绝。
- 被问“管理员怎么识别”：登录会话里有管理员标识，来源于配置化规则。

## 14. 掉线重连时，怎么恢复用户会议状态？

### 知识点
重连恢复看两件事：
- 会话是否可恢复（token 还有效）。
- 房间是否可恢复（currentMeetingId 还在）。

### 项目落地
当前恢复链路：
- token 会话存 Redis，TTL 一天。
- 新连接握手时校验 token 并绑定 userId 到连接。
- 若 token 里存在 currentMeetingId，则自动把连接加入对应会议房间。
- 掉线时会记录离线时间并触发退会相关收敛逻辑。

### 代码细节（恢复时序）
- 握手阶段先校验 token。
- 校验通过后写入用户与连接的映射关系。
- 查询会话中的 currentMeetingId，存在则自动回房。
- 断线回调会做状态更新和连接清理。

### 工程思维
- 恢复体验：尽量做到“接近无感”。
- 一致性：断线后及时清理在会关系，避免幽灵在线。
- 边界：媒体协商是否自动恢复仍依赖客户端实现。

### 追问应对
- 被问“是不是绝对无感”：不是，控制面接近无感，媒体面仍依赖客户端补协商。
- 被问“会不会脏状态”：可能，所以重连后应主动拉一次最新状态。

## 15. 踢人、拉黑和并发消息冲突时，怎么保证最终状态正确？

### 知识点
并发冲突的关键是“状态优先，消息收敛”：
- 状态写入要尽量快且可验证。
- 广播事件是通知，不应成为唯一真相。

### 项目落地
当前项目采用状态优先策略：
- 主持人操作先更新成员状态，再广播退出事件。
- 退会时会清理用户当前会议标识，避免后续被误判为在会。
- 房间广播后按正常状态过滤在线成员，空房间触发收尾。

### 代码细节（冲突场景）
- 典型冲突是“主持人踢人”和“成员主动退会”并发。
- 服务端最终以最新状态作为准，客户端按状态重渲染。
- 这是一种工程化最终一致，不是强一致锁序列。

### 工程思维
- 实时系统里先保证最终正确，比追求绝对时序更现实。
- 当前方案实现成本低、可维护，但仍有竞态窗口。
- 下一步可引入会议内 sequence 或状态版本号，进一步压缩窗口。

### 追问应对
- 被问“消息晚到怎么办”：晚到消息到达后，客户端应以最新成员状态收敛。
- 被问“怎么进一步减少竞态”：增加版本控制、顺序号和幂等校验。

## 16. 如果继续演进，你会优先做哪些升级？

### 知识点
实时系统演进建议按顺序推进：
- 先容量。
- 再一致性。
- 再可观测和运维自动化。

### 项目落地（分阶段计划）
- 阶段一（容量）：Mesh 向 SFU 渐进升级，按会议规模动态路由。
- 阶段二（一致性）：补 outbox、幂等键、死信治理、会议内 sequence。
- 阶段三（体验与运维）：在已有 Actuator、TraceId、会议指标和 Docker Compose 基础上，继续补端到端指标、弱网诊断、自动降级、灰度门禁。

项目已经具备继续演进的底座：
- 会议状态机和主持人控制。
- 邀请状态机和去重机制。
- 文件能力和路径安全。
- 管理后台和版本更新链路。

### 代码细节（为什么能演进）
- 会控逻辑集中在服务层，改策略不需要改大量入口。
- 信令转发与业务逻辑边界明确，便于替换媒体路径。
- 消息总线有统一抽象，后续可扩展更多消息中间件能力。

### 工程思维
- 升级不是“一次重构”，而是“阶段化目标 + 可回滚策略”。
- 每阶段都要有可验证指标，不靠感觉上线。
- 先补基础设施，再叠加复杂功能，长期维护成本更低。

### 追问应对
- 被问“最先做哪个”：先做 SFU 试点和观测体系，直接影响体验和容量上限。
- 被问“如何降风险”：双轨开关、灰度流量、回滚预案、关键指标门禁。

## 代码流程细节总览（统一流程章）

下面按“使用技术 -> 获取什么 -> 传给哪一层 -> 这样做的好处 -> 当前边界”统一描述核心流程。

### 流程1：登录与会话建立
- 使用技术：Spring Boot 接口层 + Redis。
- 获取什么：用户登录凭证、基础用户信息、管理员标识。
- 传给哪一层：Controller 接收后传给 Service；Service 生成 token 并写入 Redis。
- 运行态写入：token 对应用户会话对象，同时保存 userId 到 token 的映射。
- 好处：后续 HTTP 和 WebSocket 都可通过 token 快速恢复用户上下文。
- 边界：token 过期后需要重新登录，不能永久无感。

### 流程2：WebSocket 建链与鉴权
- 使用技术：Netty pipeline（HTTP 协议处理、空闲检测、鉴权、WebSocket 协议处理）。
- 获取什么：握手请求中的 token。
- 传给哪一层：鉴权 handler 先查 Redis，成功后把 userId 写入连接上下文，再传给业务 handler。
- 运行态写入：建立 userId 到 channel 的映射；如果会话有 currentMeetingId 则自动回房。
- 好处：非法连接在早期被拒绝，业务层只处理合法消息，降低复杂度。
- 边界：网络抖动下仍需客户端配合重连与状态对齐。

### 流程3：信令消息转发
- 使用技术：WebSocket 文本帧 + DTO 解析 + 消息总线（RabbitMQ）。
- 获取什么：signalType、signalData、目标用户或目标会议。
- 传给哪一层：业务 handler 解析后封装统一消息对象，交给消息处理抽象。
- 下发路径：消息总线消费后按“单播用户”或“群播会议”路由到连接层。
- 好处：跨节点同步能力统一，Netty 接入层不直接依赖 RabbitMQ 细节。
- 边界：默认语义是至少一次，当前通过 messageId 去重和业务状态收敛控制重复消费影响。

### 流程4：会议创建与入会
- 使用技术：HTTP API + Service 状态机 + Redis 运行态 + MySQL 持久化。
- 获取什么：会议信息、用户身份、入会参数。
- 传给哪一层：Controller 只做参数接收，Service 做会议状态与权限校验。
- 状态更新：写会议成员关系到 DB，并同步写会中成员运行态到 Redis。
- 消息动作：广播成员加入事件到房间。
- 好处：DB 保长期一致，Redis 保实时查询和高并发读。
- 边界：并发冲突场景依赖状态收敛，非全局强一致。

### 流程5：邀请状态机
- 使用技术：事务处理 + DB 状态字段 + WebSocket 实时通知。
- 获取什么：邀请发起人、接收人、会议信息、附加邀请文案。
- 传给哪一层：Service 先做联系人、在线、权限、去重校验。
- 状态更新：先写邀请记录为待处理，再发送通知；接受/拒绝/撤回均落状态。
- 好处：先落库后通知，避免“通知到了但状态没落地”。
- 边界：邀请链路已做去重，但系统级 outbox 仍待建设。

### 流程6：主持人控制（踢人/拉黑/结束会议）
- 使用技术：会控 Service + 成员状态枚举 + 房间广播。
- 获取什么：操作者身份、目标用户、目标状态。
- 传给哪一层：Service 先校验是否主持人或管理员，再更新状态。
- 状态更新：同步更新运行态和持久态，随后广播控制事件。
- 好处：权限和状态绑定，避免只靠前端控制。
- 边界：极端并发下仍可能出现短暂状态竞争，需版本机制进一步加强。

### 流程7：媒体状态变更广播
- 使用技术：会中成员缓存 + 房间广播消息。
- 获取什么：成员开关摄像头/麦克风意图。
- 传给哪一层：Service 更新运行态后，构造媒体变更消息广播。
- 好处：会中 UI 可实时同步，成本低于全量轮询。
- 边界：当前以成员自更新为主，主持人强制管控能力可继续补充。

### 流程8：掉线、退会与重连恢复
- 使用技术：Netty 断线回调 + Redis 会话 + 会中状态收敛。
- 获取什么：断线用户身份、当前会议标识。
- 传给哪一层：断线事件触发退会处理，更新离线时间和会中关系。
- 重连路径：新连接鉴权后恢复 user-channel 映射，满足条件时自动回房。
- 好处：控制面恢复速度快，用户体验接近无感。
- 边界：媒体协商重建仍依赖客户端；重连后建议主动做状态同步拉取。

### 流程9：聊天消息写入与分表路由
- 使用技术：分表路由工具 + MyBatis 动态表名。
- 获取什么：meetingId、messageId、发送者、消息内容、时间戳。
- 传给哪一层：Service 先按 meetingId 计算目标表，再调用 Mapper 执行写入。
- 好处：高写入场景下单表压力可控，会议内查询命中效率高。
- 边界：跨会议聚合查询复杂，后续需要离线化或汇总化处理。

### 流程10：文件上传下载与安全控制
- 使用技术：Spring Multipart + 本地文件系统 + 路径合法性校验。
- 获取什么：上传文件、当前用户会议信息、文件标识。
- 传给哪一层：Controller 先做在会权限和参数校验，再调用 Service/DAO 写文件记录。
- 下载控制：校验文件归属会议并做 canonical path 检查，拦截路径穿越。
- 好处：安全边界清晰，能防止越权和目录逃逸。
- 边界：当前是本地存储模式，跨机房和弹性扩展可继续升级到对象存储。

### 流程11：管理后台与版本更新
- 使用技术：全局拦截器 + 管理 API + 更新检查 API。
- 获取什么：管理员 token、查询条件、客户端版本信息。
- 传给哪一层：拦截器先校验登录和管理员身份，业务层执行分页、统计、强制结束会议等能力。
- 更新链路：客户端提交当前版本，服务端返回是否有新版本和下载信息。
- 好处：管理能力和用户能力隔离，版本发布有统一入口。
- 边界：当前仍是应用内控制面，后续可接入更完整发布平台。

## 一页速记（工程化快答）

1. 架构：Spring Boot 业务层 + Netty 信令层 + Redis 会话与运行态 + MySQL 持久化 + RabbitMQ 会议事件总线。
2. 技术取舍：先 Mesh 保交付速度，控制层先做稳，再演进 SFU。
3. 信令主链路：握手鉴权 -> 解析 DTO -> 统一消息封装 -> 消息总线 -> 单播/群播下发。
4. 一致性现状：RabbitMQ publisher confirm + manual ACK + 最多 3 次重试 + DLQ，邀请先落库后通知，整体至少一次语义。
5. 有序性策略：保证局部有序和状态收敛，不承诺全局强顺序。
6. 数据策略：聊天按 meetingId 哈希到 32 分表，优先保证会中写入和会议内查询。
7. 权限模型：系统管理员 + 会议主持人 + 成员状态三层叠加。
8. 重连恢复：token 会话恢复 + currentMeetingId 自动回房（控制面接近无感）。
9. 工程边界：无全链路 outbox、无全局 sequence，幂等目前以 Redis messageId 去重为主，是后续一致性建设重点。
10. 演进优先级：容量（SFU） -> 一致性（outbox/sequence/业务幂等） -> 可观测与自动化运维。
