# EasyMeeting 面试项目全解析

> 文档目标：把项目从“能跑”升级为“能讲、能答、能延展”。
> 
> 使用方式：先读“项目全景”和“各类职责”，再按“学习路线”推进，最后用“知识点拓展 + 面试模板”做输出训练。

---

## 1. 项目定位与业务价值

EasyMeeting 是一个面向中小规模在线会议场景的后端系统，核心价值不只是“开会”，而是把会议全生命周期的控制能力做完整：

- 会前：账号与权限、联系人关系、会议预约、邀请分发。
- 会中：实时入会、成员状态同步、信令转发、聊天与文件协作。
- 会后：会议信息沉淀、消息与记录可追溯、版本更新支持。

从面试表达角度，你可以将它定义为：

1. 一个以会议控制链路为核心的实时系统后端。
2. 一个融合 HTTP 业务接口与 WebSocket 实时通信的工程化项目。
3. 一个具备缓存、消息总线、关系型存储协同能力的可扩展架构。

## 2. 技术栈全景

### 2.1 基础框架

- Spring Boot：承载业务接口、依赖注入、配置治理、生命周期管理。
- MyBatis：负责数据访问与动态查询。
- HikariCP：数据库连接池。

### 2.2 实时通信与消息

- Netty + WebSocket：承载会议实时信令和在线推送。
- RabbitMQ RabbitMQ 会议事件总线：支持跨节点消息同步与解耦。

### 2.3 数据与状态

- MySQL：存放会议、用户、联系人、邀请、文件、聊天等持久化数据。
- Redis：存放登录态、令牌、临时状态与高频读写数据。

### 2.4 工程治理

- AOP 拦截：统一登录校验、权限控制。
- 全局异常处理：统一错误响应模型。
- 分层实体模型：PO / DTO / VO / Query 职责明确。

## 3. 系统架构理解（面试讲述版）

把系统拆成四个面，最容易讲清楚：

1. 接口面（HTTP）：对外提供账号、会议、联系人、文件、管理等业务能力。
2. 实时面（WebSocket）：维护长连接，完成实时信令与事件分发。
3. 数据面（MySQL + Redis）：分别承接持久化与高速状态访问。
4. 总线面（RabbitMQ）：解决消息异步化与跨节点协同。

这四个面背后的核心思想是：

- 把“请求响应型业务”与“实时推送型业务”分轨处理。
- 把“强一致持久数据”与“高频临时状态”分层存储。
- 把“本地处理”与“跨节点分发”通过消息总线解耦。

## 4. 关键链路总览

### 4.1 HTTP 业务请求链路

典型流程：

1. 客户端发起业务请求。
2. 全局拦截器执行登录与权限校验。
3. 控制层接收参数并调用服务层。
4. 服务层编排业务，必要时访问缓存与数据库。
5. 统一响应对象返回前端。
6. 异常由全局异常处理器统一收口。

面试里强调：

- 入口统一、规则统一、错误统一，这三件事决定了接口治理质量。

### 4.2 WebSocket 实时链路

典型流程：

1. 客户端建立 WebSocket 长连接并携带身份信息。
2. 服务端完成连接鉴权与上下文绑定。
3. 客户端发送会议信令或实时事件。
4. 服务端解析消息并按用户或房间路由。
5. 必要时通过消息总线同步到其他节点。
6. 目标客户端收到消息并驱动前端状态更新。

面试里强调：

- 实时系统重点不是“能发消息”，而是连接生命周期、状态一致性、跨节点协同。

## 5. 各类职责（重点）

下面是你在面试中必须讲清楚的职责分工。

| 类别 | 主要职责 | 典型输出 | 与谁协作 |
|---|---|---|---|
| Controller | 定义接口边界、接参、参数初校、返回统一响应 | 业务 API | 调用 Service |
| Service | 核心业务编排、事务控制、状态流转、规则校验 | 业务结果与状态变更 | 调用 Mapper、Redis、消息组件 |
| Mapper | 数据访问抽象、SQL 映射、动态查询 | 持久化结果 | 面向 Service |
| PO | 数据库实体映射 | 持久数据结构 | Mapper |
| DTO | 进程内传输对象，承接流程中间态 | 流程上下文 | Controller/Service/WebSocket |
| VO | 返回给前端的展示对象 | 统一响应体 | Controller |
| Query | 查询条件与分页参数封装 | 动态查询输入 | Service/Mapper |
| AOP 拦截 | 登录态校验、权限切面、通用前置规则 | 访问控制结果 | 全部业务入口 |
| 全局异常处理 | 统一错误码、错误信息、异常转义 | 标准化错误响应 | 全系统 |
| Redis 组件 | 缓存访问、会话状态、短期数据 | 高频状态读写 | Service、拦截器、实时链路 |
| WebSocket/Netty 组件 | 长连接管理、鉴权、心跳、消息路由 | 实时事件分发 | Service、消息总线 |
| 常量/枚举/工具类 | 业务语义收敛、重复逻辑复用 | 可维护基础能力 | 全系统 |

### 5.1 你可以这样概括职责边界

1. Controller 负责“收口”，Service 负责“决策”，Mapper 负责“落库”。
2. DTO/VO/PO/Query 的分层让数据在不同边界上“各司其职”。
3. AOP 与全局异常是“横向治理能力”，避免每个接口重复造轮子。
4. WebSocket 与消息总线负责“实时性和分布式协同”，是会议场景的核心能力。

## 6. 核心业务模块拆解

### 6.1 账号与权限模块

职责：登录、会话维持、权限校验、管理端身份判定。

亮点：

- 通过缓存存储登录态，减少数据库读压力。
- 通过统一拦截机制控制访问门槛。

常见追问：

- 令牌续期策略如何设计？
- 多端登录冲突怎么处理？

### 6.2 会议与预约模块

职责：创建会议、预约会议、会议状态流转、成员管理。

亮点：

- 会议状态是一个小型状态机，生命周期清晰。
- 可结合事务保证关键动作原子性。

常见追问：

- 预约冲突规则如何定义？
- 主持人权限与普通成员权限如何区分？

### 6.3 联系人与邀请模块

职责：联系人申请、关系建立、会议邀请发送与处理。

亮点：

- 关系链路与会议链路打通，便于会前组织。
- 邀请事件可通过消息总线异步分发。

常见追问：

- 如何避免重复邀请？
- 如何处理邀请超时或失效？

### 6.4 文件模块

职责：会议相关文件上传、下载、记录关联。

亮点：

- 文件元数据与会议实体关联，便于审计与追踪。
- 可扩展为对象存储方案，降低单机磁盘依赖。

常见追问：

- 文件安全扫描如何做？
- 大文件分片/断点续传怎么设计？

### 6.5 实时消息与信令模块

职责：入会广播、信令透传、在线状态变化通知。

亮点：

- 业务接口与实时链路分层，保证系统结构清晰。
- 消息总线可用于跨节点同步。

常见追问：

- 消息可靠性语义是至少一次还是恰好一次？
- 乱序与重复消费怎么收敛？

## 7. 面试亮点与可改进点

### 7.1 可以重点强调的亮点

1. 同时具备 HTTP 业务能力与 WebSocket 实时能力。
2. 缓存、数据库、消息总线三者协同，具备工程化复杂度。
3. 拦截器 + 全局异常 + 统一响应，体现治理思维。
4. 实体模型分层清晰，便于团队协作与长期维护。

### 7.2 可以主动承认并给出方案的改进点

1. 本地文件存储可升级为对象存储，增强多节点部署能力。
2. 消息可靠性可补齐死信队列、幂等键、重试可观测。
3. 实时高并发场景可补齐连接压测与容量规划模型。
4. 安全层面可加强敏感配置治理与访问审计。

面试策略：

- 不回避问题，但给出可执行改进路径，会比“只讲优点”更加分。

## 9. 详细知识点拓展（重点复习区）

下面每个专题都按“原理 - 项目落地 - 面试追问 - 加分项”组织。

### 9.1 Spring Boot 分层与治理

- 原理：分层不是形式主义，目标是降低耦合、提高可替换性。
- 项目落地：Controller/Service/Mapper 职责边界清晰，横向治理能力独立。
- 面试追问：如何避免 Service 膨胀？如何做领域化拆分？
- 加分项：引入领域服务与应用服务分离思想，进一步提升可维护性。

### 9.2 AOP 权限拦截

- 原理：把通用前置逻辑抽离，避免每个接口重复写校验。
- 项目落地：登录态和权限校验由拦截机制统一处理。
- 面试追问：AOP 与过滤器、拦截器如何选择？
- 加分项：扩展注解元数据，实现更细粒度权限策略。

### 9.3 全局异常处理与错误码体系

- 原理：异常是契约的一部分，统一处理可提升前后端协作效率。
- 项目落地：业务异常和系统异常统一转为标准响应体。
- 面试追问：如何区分可重试错误和不可重试错误？
- 加分项：建立错误码文档和可观测指标，支持故障趋势分析。

### 9.4 MyBatis 动态 SQL 与查询模型

- 原理：复杂检索应通过查询模型承载，避免参数散落。
- 项目落地：Query 对象承接条件、范围、排序、分页。
- 面试追问：动态排序如何防注入？分页深翻页如何优化？
- 加分项：引入白名单排序字段与组合索引策略说明。

### 9.5 Redis 会话与缓存一致性

- 原理：缓存提升性能，但会带来一致性窗口问题。
- 项目落地：令牌和高频状态优先走缓存，数据库作为最终持久层。
- 面试追问：缓存穿透、击穿、雪崩如何治理？
- 加分项：基于热点数据设置分级 TTL 与异步刷新策略。

### 9.6 RabbitMQ 可靠消息

- 原理：消息系统本质提供异步与削峰，可靠性需要工程补偿。
- 项目落地：通过 RabbitMQ 会议事件总线处理跨节点同步。
- 面试追问：至少一次语义下如何做幂等？
- 加分项：补充死信队列、重试退避、消息追踪链路。

### 9.7 Netty 线程模型与连接管理

- 原理：事件驱动模型适合高并发长连接场景。
- 项目落地：长连接、心跳、鉴权、消息处理形成完整流水线。
- 面试追问：如何定位连接泄漏和背压问题？
- 加分项：已补充 WebSocket 连接数、房间数、在线人数和消息成功/失败指标，可继续扩展消息延迟和堆外内存指标。

### 9.8 WebRTC 信令与 Mesh/SFU 演进思路

- 原理：信令负责协商，媒体传输策略决定系统规模上限。
- 项目落地：服务端承担信令转发和会议状态同步。
- 面试追问：什么时候从 Mesh 迁移到 SFU？
- 加分项：给出迁移分阶段方案与容量阈值判断逻辑。

### 9.9 事务边界与并发一致性

- 原理：关键业务动作需在事务内确保原子性。
- 项目落地：预约、邀请、关系变更等流程应明确提交边界。
- 面试追问：并发下如何避免重复写入？
- 加分项：唯一约束 + 幂等键 + 乐观锁联合治理。

### 9.10 分表与高写入场景优化

- 原理：高频写入表随数据增长会出现索引与锁竞争问题。
- 项目落地：会议聊天类数据可通过分表降低单表压力。
- 面试追问：分表后如何做跨分片查询与归档？
- 加分项：冷热分层、归档策略、离线分析链路。

### 9.11 安全治理

- 原理：安全不是单点功能，是配置、鉴权、审计、传输的组合。
- 项目落地：统一鉴权、统一异常、基础权限控制已具备。
- 面试追问：敏感配置如何管理？接口防重放如何做？
- 加分项：引入密钥托管、审计日志、请求签名策略。

### 9.12 可观测性与故障排查

- 原理：没有可观测性就没有稳定性。
- 项目落地：日志已接入 TraceId，关键链路可携带 meetingId、userId、messageId，配合 Actuator 暴露健康检查和会议运行指标。
- 面试追问：如何定位“用户反馈无法入会”的问题？
- 加分项：建立指标三件套（日志、指标、链路），形成排查 SOP。

## 10. 面试表达模板（直接可用）

### 10.1 一分钟版本

EasyMeeting 是一个实时会议后端系统，采用 Spring Boot 承载业务接口，Netty + WebSocket 承载实时信令，MySQL 做持久化，Redis 做会话与高频状态缓存，并通过 RabbitMQ 会议事件总线处理跨节点同步。项目的核心价值是把会议生命周期做完整，包括会前预约和邀请、会中实时协作、会后数据沉淀。我的主要贡献聚焦在分层治理、实时链路稳定性和业务规则落地。

### 10.2 三分钟版本

项目架构上我把它拆成四个面：接口面、实时面、数据面、总线面。接口面负责账号、会议、联系人、文件等请求响应型业务；实时面负责长连接、心跳、鉴权和消息路由；数据面由 MySQL 和 Redis 分工协作；总线面解决跨节点事件同步。工程治理上，通过 AOP 做统一权限拦截，通过全局异常处理统一错误语义，通过 PO/DTO/VO/Query 分层控制数据边界。这个项目的难点在于既要保证业务接口规范性，又要保证实时链路稳定性，所以我重点关注了连接管理、状态一致性和消息可靠性。

### 10.3 十分钟版本

先讲业务全景，再讲两条核心链路：

1. HTTP 业务链路：拦截校验、控制层收口、服务层决策、缓存和数据库协同、统一响应与异常收口。
2. 实时信令链路：连接鉴权、会议信令解析、按用户/房间路由、跨节点同步、客户端状态收敛。

然后讲工程化亮点：

1. 分层明确，职责清晰，便于扩展。
2. 实时链路独立，适配会议场景。
3. Redis 状态缓存与 RabbitMQ 事件总线配合，具备横向扩展基础。

最后讲改进方案：

1. 一致性继续增强（outbox、会议内 sequence、业务幂等表）。
2. 存储能力增强（对象存储、冷热分层）。
3. 可观测性增强（指标体系和故障演练）。

## 11. 高频追问与答题要点

1. 为什么不用单纯的请求响应架构？
结论：会议场景强依赖实时协作，必须有长连接链路承接低延迟事件。

2. 为什么需要 Redis？
结论：会话和高频状态读取明显高于写入，缓存能显著降低数据库压力。

3. 消息总线的价值是什么？
结论：把实时事件异步化和跨节点化，避免服务间强耦合。

4. 如何保证系统稳定？
结论：分层治理 + 统一异常 + 心跳机制 + RabbitMQ 重试/DLQ + TraceId 日志与运行指标。

5. 你做了哪些可扩展设计？
结论：职责边界清晰、消息总线固定使用 RabbitMQ、实时链路与业务链路解耦。
